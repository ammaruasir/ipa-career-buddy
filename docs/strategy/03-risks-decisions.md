# المخاطر والقرارات المفتوحة

> **الغرض:** هذه قرارات استراتيجية ومخاطر تقنية يجب حسمها بالتوازي مع بدء التنفيذ، لتجنّب إعادة العمل لاحقاً.

---

## القرارات الاستراتيجية المؤجَّلة

### قرار 1: مصير HR Pipeline

**الوضع:** البنية الحالية تحتوي:
- `job_vacancies` table + `job_applications` table
- `HiringPipeline.tsx` (Kanban للمراحل: applied → screening → interviewing → offered → hired → rejected)
- `evaluate-interview` يحدّث `job_applications.status` تلقائياً
- HR Dashboard كامل

**السؤال:** هل IPA يدير برامج تدريب فقط، أم برامج تدريب **وتوظيف فعلي** لجهات حكومية شريكة؟

**الخيارات:**

| الخيار | المعنى | الأثر |
|--------|--------|-------|
| **(أ) تدريب فقط** | حذف كل بنية HR | -2 جدول، -1 صفحة، تبسيط جوهري للـ codebase |
| **(ب) ثنائية: تدريب + توظيف** | الإبقاء على البنية، فصلها بـ `interviews.mode` | البنية الحالية مدعومة كما هي + الإضافات الجديدة |
| **(ج) feature flag** | تعطيل HR pipeline افتراضياً، تفعيلها للجهات الشريكة | تعقيد إضافي بدون منفعة واضحة (لا يُنصح به) |

**التوصية الافتراضية:** (ب) — احتفظ بـ HR pipeline، اعزلها بـ `interviews.mode`. هذا الخيار الذي تنبني عليه خطة P0 الحالية.

**القرار يجب أن يُحسَم خلال:** الأسبوع 1 من بدء التنفيذ.

### قرار 2: استضافة البيانات داخل المملكة (PDPL)

**الوضع:** المنصة حالياً على Supabase Cloud. Supabase لا يوفّر **منطقة سعودية** (Riyadh region) حتى الآن.

**المتطلّب القانوني المحتمل:** PDPL السعودي (نظام حماية البيانات الشخصية) يُلزم بـ "الاستضافة داخل المملكة" للبيانات الحساسة، وبيانات الطلاب الحكوميين قد تقع في هذه الفئة.

**الخيارات:**

| الخيار | الجاهزية | التكلفة |
|--------|----------|---------|
| **Supabase Cloud (الحالي)** | جاهز فوراً | منخفضة، لكن **مخاطرة قانونية** |
| **Supabase Self-hosted على STC Cloud** | يحتاج devops، 2–4 أسابيع | متوسطة، يحلّ مشكلة الموقع |
| **Microsoft Azure Saudi (الرياض)** | يحتاج هجرة كاملة | عالية، 4–8 أسابيع |
| **Oracle Cloud Saudi** | يحتاج هجرة + إعادة بناء بعض المكوّنات | عالية جداً |

**التوصية:** ابدأ في pilot على Supabase Cloud (موافقات صريحة من المشاركين + بيانات اختبار)، لكن **اتخذ قرار البنية النهائية قبل إطلاق إنتاج عام**.

**القرار يجب أن يُحسَم خلال:** قبل المرحلة 3 من pilot (أي قبل ~12 أسبوعاً).

### قرار 3: نموذج الشراكة مع IPA

**السؤال:** هل المنصة:
- (أ) **منتج معتمَد رسمياً من IPA** (شراكة استراتيجية، قد تشمل تكامل مع أنظمة IPA الداخلية)؟
- (ب) **أداة طرف ثالث** تستهدف طلاب IPA دون موافقة رسمية؟

**الأثر:**

| الجانب | (أ) معتمَد | (ب) طرف ثالث |
|--------|------------|--------------|
| المحتوى | وصول لخبراء IPA لكتابة 200+ سؤال | يجب توظيف خبراء خارجيين |
| التكامل | Moodle/أنظمة IPA الداخلية ممكنة | فقط نظام self-contained |
| المصداقية | شعار IPA + اعتماد رسمي | تسويق مستقل |
| الامتثال | بيئة compliance أوضح | عبء الامتثال على المنتج |

**القرار يجب أن يُحسَم خلال:** قبل بدء P0.5 (محتوى الأسئلة).

### قرار 4: تكلفة LLM و TTS

**النماذج الحالية:**

| الاستخدام | النموذج | التكلفة التقديرية لكل جلسة |
|----------|---------|--------------------------|
| `evaluate-interview` | محرك واكب (وحدة نصية كاملة) | ~$0.30 - $0.60 |
| `analyze-resume` | محرك واكب (وحدة متعدّدة الوسائط) | ~$0.05 - $0.10 |
| `career-guidance` | محرك واكب (وحدة متعدّدة الوسائط) | ~$0.10 |
| `chat` | محرك واكب (وحدة نصية مختصرة) | ~$0.05 - $0.15 |
| TTS (وحدة الصوت في محرك واكب) | متوسط 25 دقيقة/جلسة | **~$7.50** |

**وحدة الصوت هي أكبر تكلفة تشغيلية**، ليس LLM.

**تقدير شهري لـ 100 طالب × 5 جلسات:**
- LLM: ~$200/شهر
- TTS: **~$3,750/شهر**

**خيارات تخفيض TTS:**

| الخيار | التوفير | الأثر |
|--------|--------|------|
| استخدام voice أقل جودة في practice | 30-40% | أقلّ واقعية |
| TTS بديل (موفّر ثانوي) | 60-70% | جودة عربية أدنى لكن مقبولة |
| TTS مفتوح المصدر داخلي | 90%+ | يحتاج GPU، جودة أقل |
| تقليل أسئلة practice من 8 إلى 5 | 37% | منطقي للممارسة |

**التوصية:** نمذجة دقيقة قبل pilot المرحلة 2. ضع budget cap في `system_settings` (مثل `max_tts_minutes_per_user_per_month`).

## المخاطر التقنية

### مخاطرة 1: enum mismatch في الإنتاج

**الخطر:** عند تنفيذ P0.0، إذا كانت بيانات الإنتاج تحتوي صفوفاً بدور `'candidate'`، فإن الـ migration قد تفشل.

**التخفيف:**
- اختبار الـ migration على نسخة من قاعدة الإنتاج أولاً
- نشر الـ migration في نافذة صيانة (15 دقيقة كافية)
- backup للقاعدة قبل التنفيذ

### مخاطرة 2: `evaluate-interview` يكسر في practice mode

**الخطر:** الكود الحالي يفترض دائماً وجود `job_vacancy`. إذا نشرت P0.1 بدون P0.1 الكاملة (الـ Edge Function أيضاً)، practice mode لن يعمل.

**التخفيز:**
- لا تنشر migration الـ mode قبل تحديث الـ Edge Function
- اختبر practice mode بنفس فحص E2E قبل النشر

### مخاطرة 3: جودة RTL PDF في CV Builder

**الخطر:** Arabic ligatures + bidi + tables = نقاط ضعف معروفة في كل مكتبات PDF.

**التخفيف:**
- spike يومان قبل الالتزام (انظر P0.4)
- اختبار الناتج على 5 PDF readers (Adobe، Foxit، macOS Preview، Chrome، Firefox)
- خطّة backup: إذا فشل react-pdf، انتقل لـ Headless Chromium

### مخاطرة 4: تكلفة Edge Functions غير المتوقّعة

**الخطر:** P0.2 (coaching لكل إجابة) قد يضاعف تكلفة LLM في الجلسة.

**التخفيف:**
- استخدم وضع التدريب المختصر من محرك واكب للـ coaching (أرخص بنحو 4x من الوضع الكامل)
- خيار "on-demand": coaching افتراضياً مختصر، الطالب يضغط "اشرح بعمق" للحصول على تحليل كامل
- caching: إذا تم coaching نفس الإجابة من قبل (بـ hash)، أعد النتيجة من cache

### مخاطرة 5: حدود TTS في محرك واكب للذكاء الاصطناعي

**الخطر:** خطّة Free/Starter محدودة بـ 30,000 حرف/شهر. عند 100 طالب نشط، التجاوز محتمل.

**التخفيف:**
- ترقية الاشتراك قبل pilot المرحلة 2
- مراقبة `system_settings.tts_usage_current_month`
- fallback لـ browser native TTS عند تجاوز الحدّ (جودة أدنى لكن لا توقّف)

### مخاطرة 6: شخصية `panel_pressure` تسبّب ضائقة نفسية

**الخطر:** متدرّب حساس قد يصاب بنوبة قلق أو يفقد الثقة. مخاطرة قانونية على معهد حكومي.

**التخفيف:**
- **شرط blocking:** approval مكتوب من اختصاصي نفسي قبل التفعيل
- safety_gate: كشف تلقائي لمؤشّرات الضائقة (نبرة، توقف طويل، انسحاب) → تليين فوري
- زر "إيقاف" دائم وبارز في UI
- موافقة صريحة منفصلة قبل بدء الجلسة
- لا تفعيل قبل P1، ولا تفعيل في pilot

### مخاطرة 7: التحيّز اللهجي في تقييم LLM

**الخطر:** محرك واكب للذكاء الاصطناعي قد يفضّل المتحدّثين بالفصحى على لهجات (نجدي/حجازي/جنوبي).

**التخفيف:**
- في pilot المرحلة 2: عيّنة 20 طالب من مناطق مختلفة، قياس فرق الدرجات
- إذا انحياز >5 نقاط: ضبط البرومبت ("احكم على المحتوى لا اللهجة") + إضافة بشري للمراجعة
- documentation شفّاف في "كيف يعمل التقييم"

### مخاطرة 8: data leak عبر `.env` في الـ repo

**الوضع الحالي:** `.env` مُتتبَّع في git (موجود في latest commit). إذا كان يحتوي مفاتيح Supabase Service Role، فهي مكشوفة لأي شخص لديه clone للـ repo.

**الإجراء العاجل:**
1. فحص محتوى `.env`
2. إذا كان يحتوي مفاتيح حقيقية: rotation فوري للمفاتيح في Supabase Dashboard
3. إضافة `.env` لـ `.gitignore`
4. حذف `.env` من تاريخ git عبر `git filter-branch` أو `git-filter-repo`
5. إعادة push (force) إذا كان الـ repo private، أو إنشاء repo جديد إذا كان public
