## 1. السياق (Context)

معهد الإدارة العامة (IPA) هو ذراع التدريب الحكومي في المملكة العربية السعودية منذ 1961، ومهمته إعداد الكوادر للقطاع العام بما يتوافق مع رؤية 2030. المنصة الحالية بُنيت في الأصل كأداة توظيف (HR pipeline + interview evaluations) — لكن الواقع أن المستخدم النهائي ليس مرشحاً يتقدّم لوظيفة، بل **متدرّب** يحتاج بيئة آمنة للممارسة + تغذية راجعة تعليمية + أدوات سيرة ذاتية + محاكاة مقابلات شبه حقيقية.

النتيجة: كل ميزة في المنصة اليوم مصمّمة لـ "تقييم نهائي" بدلاً من "تدريب تكويني"، وهذا يقلب جوهر العلاقة بين الطالب والمنصة. هذه الوثيقة ترسم خارطة طريق لإعادة تموضع المنصة بالكامل.

## 2. لقطة الحالة الراهنة

### نقاط القوة التي يجب الحفاظ عليها

- **واجهة عربية كاملة RTL** + برومبت ذكاء اصطناعي عربي.
- **شخصية مقابِل مخصّصة ثقافياً** (Avatar بزي سعودي — ثوب/عباءة) في `src/components/interview/AIAvatarScene.tsx`.
- **أصوات عربية أصلية** عبر ElevenLabs (هيثم/سناء) — `supabase/functions/elevenlabs-tts/index.ts`.
- **محرّك تقييم متطوّر**: DISC + 5 أبعاد + توصية نهائية في `supabase/functions/evaluate-interview/index.ts`.
- **بنية مقابلة حيّة مرحلية** (INTRO → CORE → CLOSING → END) مع كشف صمت ومزامنة TTS — `src/hooks/useLiveInterview.ts` + `src/components/interview/LiveInterview.tsx`.
- **تسجيل مرن** مع checkpoint كل 20 ثانية، ومقاومة لانقطاع الاتصال.
- **حماية ضد الغش** متعددة الطبقات: تبديل التبويبات + كاميرا خلفية + تحليل إطارات — `src/hooks/useAntiCheat.ts` + `useCheatCamera.ts` + `supabase/functions/analyze-video/`.
- **RLS صارم** على جميع الجداول + RBAC نظيف (candidate/hr/admin).

### الثغرات الحرجة (مرتّبة حسب الأثر التعليمي)

| # | الثغرة | الأثر على التدريب |
|---|--------|-------------------|
| 1 | كل مقابلة تُسجَّل كتقييم رسمي ويراها HR | الطالب لا يجرؤ على الفشل = لا يتعلّم |
| 2 | لا توجد تغذية راجعة لكل إجابة (per-answer) | الطالب يعرف "أنه ضعيف" دون أن يعرف "لماذا وكيف يحسّن" |
| 3 | لا تعليم لمنهجية STAR / السلوكي | لا إطار معرفي ينقل المهارة |
| 4 | تحليل السيرة الذاتية = استخراج فقط بدون تقييم/اقتراحات | لا يتعلّم الطالب كيف يحسّن سيرته |
| 5 | لا يوجد منشئ سيرة ذاتية (CV Builder) | فجوة كاملة في الخدمة المعلن عنها |
| 6 | بنك الأسئلة عام بدون مسارات IPA | محتوى غير ملائم لجدارات والقطاع العام |
| 7 | لا يوجد منهج تدريبي / مسار تعلّم / تتبّع تقدّم | تجربة منعزلة بلا تطوّر واضح |
| 8 | شخصية مقابِل واحدة فقط | لا تنوّع في محاكاة المقابلات الحقيقية |
| 9 | لا توجد طبقة مدرّس / دفعة (cohort) | المعهد لا يستطيع تشغيل برامج جماعية |
| 10 | لا يوجد فصل عربي/إنجليزي حقيقي | المعهد يدرّس باللغتين |
| 11 | لا مقارنة بين السيرة الذاتية والإجابات | فرصة تعليمية ضائعة |
| 12 | لغة الواجهة "توظيفية" (مرشح، شواغر) | تشتّت الرسالة التدريبية |

## 4. خارطة الطريق المُوصى بها (P0 → P1 → P2)

### الطبقة P0 — أساسية (لا يمكن تسميتها "منصة تدريب" بدونها)

#### P0.1 — فصل وضع التدريب عن وضع التقييم

**الفكرة:** لا تذهب كل مقابلة إلى HR. الافتراضي = `practice` (خاص بالطالب)، وضع `assessment` فقط هو الذي يصل لمسؤول التوظيف.

**التغييرات الجوهرية:**
- إضافة عمود `mode` و `visibility` لجدول `interviews`.
- تحديث RLS: HR ترى فقط `visibility = 'hr'`.
- في `evaluate-interview`: في وضع `practice`، يُكتب التقييم لكن **لا يصل إشعار HR ولا يُحدّث `job_applications`**.
- استخدام نموذج أرخص (Gemini Flash) في `practice` ونموذج متقدّم (GPT‑4.1) في `assessment`.

**لماذا الأولوية القصوى:** هذا الحاجز الأكبر بين "أداة مراقبة" و"أداة تعلّم".

---

#### P0.2 — تغذية راجعة لكل إجابة + كوتشينج STAR

**الفكرة:** بعد كل مقابلة، يرى الطالب لكل سؤال:
- نسبة تغطية STAR (Situation / Task / Action / Result)
- إعادة كتابة محسّنة لإجابته بلغة عربية رسمية
- إجابة نموذجية ("هكذا تبدو الإجابة القوية")
- مواضع كلمات الحشو (filler words) بدمج timestamps من Whisper
- مقطع صوتي/فيديو لإجابة الطالب نفسه قابل للتشغيل من النقطة

**التغييرات:**
- `responses.coaching` JSONB جديد
- وظيفة جديدة `coach-response` أو تمرير ثانٍ داخل `evaluate-interview`
- استثمار `system_settings.filler_words` الموجود مسبقاً
- إعادة استخدام `VideoPlayback.tsx` لعرض مقاطع الإجابة

**لماذا حاسمة:** بدون هذا، الطالب يحصل على رقم 0–100 ولا يعرف كيف يتحسّن.

---

#### P0.3 — مُقيِّم السيرة الذاتية بدلاً من المستخرج

**الفكرة:** ترقية `analyze-resume` ليُعطي:
- درجات لكل قسم (Contact / Summary / Experience / Education / Skills / Achievements / Language Quality)
- نقاط الضعف بأمثلة محدّدة من السيرة الفعلية
- إعادة كتابة عربية لكل بند ضعيف ("نسختك" مقابل "نسخة محسّنة" مع السبب)
- محاذاة مع وظيفة مستهدفة عند الطلب (`target_role`)
- مراعاة المعايير السعودية: التواريخ الهجرية، تنسيق العنوان، خدمة العلم، رابط جدارات

**التغييرات:**
- جدول جديد `cv_documents` (نسخ مع تقييمات)
- تمديد schema الخاص بـ tool calling في `analyze-resume/index.ts`
- صفحة جديدة `CVReview.tsx` مع radar + اقتراحات قابلة للتطبيق

---

#### P0.4 — منشئ سيرة ذاتية عربي (CV Builder)

**الفكرة:** نموذج خطوة-بخطوة (Stepper):
1. البيانات الشخصية → 2. الملخّص → 3. الخبرات → 4. التعليم → 5. المهارات → 6. الشهادات → 7. المعاينة → 8. التصدير PDF

3 قوالب IPA-styled: محافظ / حديث / تنفيذي. توليد ذكي للنقاط بصيغة STAR-bullet عبر وظيفة `generate-cv-bullets`.

**اعتبارات تقنية حرجة:**
- Arabic RTL PDF: استخدام `@react-pdf/renderer` مع `bidi-js` أو خادم headless Chromium
- خط Noto Naskh Arabic لضمان جودة الطباعة
- بنك تخزين: استخدام `resumes/` bucket الموجود مع subpath `drafts/{user_id}/`
- خيار ثنائي اللغة (عربي/إنجليزي) منذ البداية

**لماذا أعلى نقطة استثمار:** فجوة كاملة في الخدمة، وأكثر ميزة سيقدّرها الطلاب.

---

#### P0.5 — بنك أسئلة IPA / جدارات

**الفكرة:** تمديد `question_templates` بأعمدة جديدة:
- `track` (HR / IT / المالية / الإدارة العامة / إدارة المعلومات / المكتبات / ...)
- `competency` (اتخاذ القرار / خدمة المواطن / العمل الجماعي / الأخلاقيات / التحوّل الرقمي / ...)
- `is_scenario` (سؤال موقفي مقابل تقليدي)
- `gov_context` (هل يتطلّب فهماً للسياق الحكومي السعودي)
- `model_answer_ar` (إجابة نموذجية)
- `star_rubric` (معايير تقييم STAR لكل سؤال)

**المحتوى:** 200+ سؤال مُعتمَد من خبراء IPA — حوكمة المحتوى عنصر حاسم (P1.5).

---

#### P0.6 — طبقة المدرّس والدفعات (Cohort Layer) — مطلوبة الآن

**الفكرة:** المعهد ليس فرداً — هو مؤسسة. لا يمكن تشغيل برنامج تدريبي بدون مدرّسين ودفعات.

**التغييرات:**
- تمديد enum `app_role` بإضافة قيمة `instructor`
- جداول جديدة:
  - `cohorts (id, name, track, start_date, end_date, instructor_id, ...)`
  - `enrollments (cohort_id, student_id, status)`
  - `assignments (cohort_id, type [interview/cv/quiz], target_track, due_at, requirements)`
  - `instructor_feedback (interview_id|response_id, instructor_id, annotation_text, timestamp)`
- صفحات جديدة:
  - `InstructorDashboard.tsx` — نظرة عامة على الدفعات
  - `CohortDetail.tsx` — قائمة الطلاب + متوسط الأداء + تتبّع المهام
  - `AssignmentView.tsx` — مراجعة المهمة وإضافة تعليقات منسوبة لزمن محدد في الفيديو/الصوت
- RLS: `instructor` يرى فقط مقابلات/سير ذاتية طلاب دفعته
- توسيع `DashboardRouter.tsx` بفرع جديد

**لماذا الآن:** بدونها، المنصة "ذاتية الخدمة" فقط — وهذا لا يخدم الطلاب المسجّلين في برامج IPA.

### الطبقة P1 — عالية الأثر (المرحلة الثانية)

- **P1.1 — مسار التعلّم وخريطة المهارات الحرارية:** جدول `skill_progress` view يجمع `evaluations.detailed_scores` عبر الزمن → heatmap 6 أبعاد × N مقابلة + قائمة تحقّق "الجاهزية للمقابلة النهائية" (≥75 على 5 أبعاد عبر 3 جلسات). يُعاد استخدام recharts.
- **P1.2 — شخصيات مقابِل متعدّدة:** Personas في `system_settings`:
  - `friendly_hr` (الحالي)
  - `technical_lead` (تقني صارم)
  - `panel_pressure` (ضغط + أسئلة ملاحقة قاسية — مع مفتاح أمان: إيقاف عند رصد ضائقة)
  - `senior_government` (مقابِل تنفيذي بأسلوب حكومي رسمي)
  - `library_specialist` (لمسارات معيّنة بـ IPA)

  لكل شخصية: قالب برومبت + voice_id + Avatar variant.
- **P1.3 — محاذاة السيرة الذاتية ↔ الإجابات:** بعد المقابلة، رسالة من نوع "ادّعيت في سيرتك أنك تتقن X لكن لم تذكره في أي إجابة" أو "أظهرت مهارة Y لم تذكرها — أضفها لسيرتك". تمرير ثانٍ بعد `evaluate-interview` يقارن `profiles.resume_skills` بنصوص الإجابات.
- **P1.4 — حوكمة بنك الأسئلة وصعوبة تكيّفية:**
  - `question_templates.status` (draft/review/approved/retired) + `reviewed_by` + `review_notes`
  - معايرة الصعوبة بطريقة IRT-lite: p-value لكل سؤال من `responses` التاريخية
  - الاختيار التكيّفي: إذا الطالب ينجح في الصعب، نقفز؛ إن فشل في السهل، نتراجع.
- **P1.5 — تفعيل المسار الإنجليزي بالكامل:** استبدال `localStorage app_lang` stub بـ `react-i18next` كامل + قوالب برومبت إنجليزية + أصوات ElevenLabs إنجليزية + قوالب CV إنجليزية. أوتوماتيكي RTL/LTR.
- **P1.6 — لغة التدريب بدلاً من لغة التوظيف:** مراجعة شاملة لنصوص الواجهة: "مرشح" → "متدرّب"، "الشواغر" → "المسارات الوظيفية المستهدفة"، "تم إكمال المقابلة بنجاح" → "أحسنت! الخطوة التالية: ...". فحص لكل صفحة في `src/pages/`.

## 5. أفضل الممارسات لمنصات تدريب المقابلات بالذكاء الاصطناعي

1. **افصل التكويني عن النهائي (Formative vs Summative):** الممارسة لا تُصرَف للمقيّمين أبداً؛ فقط جولات "النهائية" المُسمّاة صراحة. هذا هو الفرق بين أداة تدريب وأداة مراقبة.
2. **علّم إطاراً، لا درجة فقط:** STAR (أو CAR/PAR) مع مقاييس تغطية لكل عنصر تتفوّق على رقم 0–100 في كل مرة.
3. **اعرض، لا تقل:** بجانب نسخة الطالب، اعرض "هكذا تبدو الإجابة القوية" — المقارنة أقوى إشارة تعليمية.
4. **نصوص + تنقّل بالنقر:** transcripts مع timestamps، نقرة على كلمة → تشغيل الصوت من نفس اللحظة، خريطة حرارية لكلمات الحشو والوقفات.
5. **خريطة مهارات عبر الزمن:** Radar/Heatmap طولي حسب الكفاءة يكشف الجمود — درجة واحدة لا تفعل ذلك.
6. **حوكمة بنك الأسئلة:** كل سؤال له حالة، مالك، مراجِع، إجابة نموذجية، معايرة صعوبة؛ يُسحَب المتقادم؛ يُراجَع التحيّز الثقافي ربعياً.
7. **منع التحيّز في تقييم LLM للعربية:** اخفِ الاسم/الجنس/الصورة/مؤشّرات اللهجة عن المقيّم النموذج؛ عايِر مقابل عيّنة بشرية مرجعية؛ اختبر بشكل دوري التحيّز للهجات (نجدي/حجازي/خليجي/فصحى).
8. **نظافة PDPL (نظام حماية البيانات الشخصية السعودي):** موافقة صريحة للتسجيل (منفصلة عن شروط الاستخدام)، احتفاظ بالبيانات بحسب الغرض (90 يوماً للممارسة / 12 شهراً للتقييم)، الاستضافة داخل المملكة، واجهة "حق المحو"، حماية بيانات القاصرين.
9. **ملاءمة ثقافية:** تجنّب الافتراضات الأمريكية (الترويج العدواني للذات). علّم نبرة الحكومة السعودية: التواضع، احترام التسلسل الإداري، الاتساق مع ركائز رؤية 2030. صمّم rubrics بشراكة مع خبراء IPA.
10. **RTL وإمكانية الوصول من اليوم الأول:** صحّة bidi في PDF/charts، تسميات قارئ شاشة بالعربية، تباين للضعاف، تنقّل لوحة المفاتيح، ترجمات للفيديو، نصوص للصوت.
11. **طبقة المدرّس والدفعة:** منصات التدريب بدون لوحات معلّمين تصل لسقف التعلّم سريعاً. التعليق البشري على النصوص يتفوّق على AI-only.
12. **سلامة الممارسة:** شخصيات الضغط/اللجنة لها قاطع كهرباء (كشف ضائقة → تليين) وزر إيقاف للطالب؛ عدّاد تبديل التبويبات يخصّ وضع التقييم فقط، لا التدريب.
13. **تقييم متدرّج التكلفة:** نموذج رخيص (Gemini Flash) للممارسة، نموذج متقدّم (GPT‑4.1) للتقييم؛ خزّن التقييمات؛ لا تُعد تشغيل النموذج على نسخ متطابقة.
14. **شفافية المعايير:** انشر معايير التقييم للطلاب — الغموض يدمّر الثقة والتعلّم.

## 6. خطة التحقّق والتجريب (Validation)

قبل الالتزام بكامل خارطة الطريق، نقترح تجربة تجريبية بثلاث مراحل:

### المرحلة 1 — تحقّق المفهوم (Concept Validation) — 2 أسبوع

- **النشاط:** ورش مع 3–5 من خبراء IPA + 10 طلاب من برامج جارية
- **الهدف:** التحقّق من أن تغذية STAR + منشئ السيرة الذاتية + وضع التدريب هي أعلى احتياجات الطالب فعلاً
- **الأدوات:** نماذج Figma + Loom walkthroughs بدون كود
- **معيار النجاح:** ≥80% من المشاركين يؤكّدون أن هذه الميزات تحلّ مشاكل حقيقية

### المرحلة 2 — تجربة P0 المغلقة (Closed Pilot) — 4 أسابيع

- **النشاط:** بناء P0.1 + P0.2 + P0.3 (الأرخص للتنفيذ) ونشرها لـ 30 طالب من دفعة واحدة في IPA
- **الأدوات الميدانية:**
  1. مقارنة قبل/بعد: درجات تقييم لنفس الطلاب في مقابلة مكتوبة قبل التدريب وبعد 5 جلسات تدريب
  2. NPS (Net Promoter Score) أسبوعي
  3. تحليل سجل الاستخدام: عدد جلسات الممارسة، معدل العودة، الوقت في الجلسة
- **معيار النجاح:**
  - تحسّن متوسّط ≥15 نقطة في الدرجة الإجمالية بين أول جلسة وخامس جلسة
  - معدّل عودة ≥60% أسبوعياً
  - NPS ≥40

### المرحلة 3 — اختبار التوسّع (Scale Test) — 6 أسابيع

- **النشاط:** بناء P0.4–P0.6 + طبقة المدرّس، وتجربة كاملة مع 3 دفعات (≈90 طالب) و3 مدرّسين
- **معيار النجاح:**
  - الأدوات المكتمَلة (مقابلات + سيرة ذاتية): ≥70% من الطلاب أنهوا 5 جلسات + سيرة مُحدّثة
  - رضا المدرّسين: ≥75% يقولون "تساعدني فعلاً في تقييم دفعتي"
  - ≥1 شركة/جهة حكومية تأخذ سيرة ذاتية واحدة على الأقل من المنصة بصيغتها الجديدة

### اختبارات تقنية لازمة عند تنفيذ أي P0

- اختبار End-to-End: طالب يكمل دورة كاملة (تسجيل → سيرة ذاتية → تدريب 3 مرات → تقييم نهائي → مدرّس يعلّق)
- اختبار RLS: محاولة وصول طالب لمقابلات طالب آخر / مدرّس لدفعة أخرى — يجب أن تفشل
- اختبار حمل ElevenLabs (تكلفة + latency) عند تشغيل 50 جلسة متزامنة
- اختبار جودة OCR للسيرة الذاتية الممسوحة (scanned PDF)
- اختبار جودة RTL في PDF المُولَّد على أنظمة تشغيل مختلفة
- اختبار التحيّز اللهجي: عيّنة من 20 طالب من مناطق مختلفة، فحص ما إذا كانت الدرجات منحازة لمتحدّثي الفصحى

## 7. مخاطر وأسئلة مفتوحة للنقاش لاحقاً

1. **الموافقة الرسمية من IPA:** هل سيكون هذا منتجاً معتمداً رسمياً من المعهد، أم أداة من طرف ثالث تستهدف طلابه؟ هذا يحدّد متطلّبات الامتثال والتكامل (Moodle/أنظمة IPA الداخلية).
2. **تكلفة LLM:** التغذية الراجعة المفصّلة لكل إجابة باهظة على GPT‑4.1. القرار: متى نستخدم Flash مقابل GPT‑4.1، وهل نسمح للطالب باستدعاء "اشرح إجابتي بعمق" يدوياً (on-demand) بدلاً من تشغيلها افتراضياً؟
3. **حقوق المحتوى:** بنك الأسئلة و"الإجابات النموذجية" — من يملكها؟ هل يمكن لـ IPA مشاركتها مع طلابها أم تبقى حصرية للمنصة؟
4. **حدود وضع الضغط:** شخصية `panel_pressure` لها مخاطر نفسية — تحتاج مراجعة من اختصاصي نفسي قبل التفعيل.
5. **جدارات API:** هل توجد واجهة برمجية لمنصة جدارات يمكن التكامل معها لاستيراد متطلّبات الوظائف الحقيقية؟ هذا يقلّل عبء صيانة بنك الوظائف.
6. **سياسة الاحتفاظ بالتسجيلات:** تسجيلات المقابلات تحتوي صور وأصوات الطلاب — أعلى مستوى حساسية PDPL. السياسة المقترحة: حذف تلقائي بعد 90 يوماً للممارسة، تشفير عند الراحة، تصدير مشفّر للأرشيف.
7. **هل نحتفظ ببنية HR كاملةً؟** المنصة الحالية فيها HR pipeline كامل (`HiringPipeline.tsx` كانبان). هل نزيلها كلياً لأن IPA معهد تدريب، أم نحتفظ بها كمسار اختياري للجهات الحكومية التي تستخدم المنصة فعلاً للتوظيف؟ (التوصية: احتفظ بها لكن اعزلها خلف feature flag في `system_settings`).

## 8. الخلاصة التنفيذية

المنصة الحالية **منتج توظيف ممتاز فنياً** (Arabic-first، avatar ثقافي، حماية ضد الغش، تقييم DISC، تسجيل مرن). لتصبح **منصة تدريب IPA حقيقية**، تحتاج إلى:

1. **انقلاب فلسفي:** التدريب أولاً، التقييم استثناء. (P0.1)
2. **تحوّل في التغذية الراجعة:** من رقم إلى كوتشينج تعليمي لكل إجابة. (P0.2)
3. **إكمال سلسلة الخدمة:** سيرة ذاتية كاملة (تقييم + بناء + محاذاة). (P0.3 + P0.4 + P1.3)
4. **محتوى متخصّص:** أسئلة سعودية حكومية بحوكمة مهنية. (P0.5 + P1.4)
5. **طبقة مؤسسية:** مدرّسون ودفعات وتقدّم. (P0.6 + P1.1)
6. **لغة وموقف تدريبيان:** كل نص في الواجهة يقول "أنت تتعلّم"، لا "أنت تُقيَّم". (P1.6)

**ترتيب التنفيذ المُوصى به للمرحلة الأولى:**

P0.1 (وضع التدريب) → P0.2 (تغذية راجعة لكل إجابة) → P0.6 (طبقة المدرّس) → P0.5 (بنك أسئلة IPA) → P0.3 (تقييم السيرة) → P0.4 (منشئ السيرة).

السبب: المنتج بحاجة "هيكل التعلّم" قبل "أدوات التعلّم". الفصل بين التدريب والتقييم + طبقة المدرّس + التغذية الراجعة لكل إجابة هي الـ"عظام"؛ السيرة الذاتية وبنك الأسئلة هي الـ"عضلات".

> **هذه وثيقة استشارية:** لا أكواد ستُكتب ولا تعديلات على المنصة ستُنفّذ بناءً عليها مباشرة. الخطوة التالية المقترحة: مراجعتها مع فريق IPA + فريق المنتج، ثم تحويل بنود P0 المتفق عليها إلى خطط تنفيذية مفصّلة منفصلة (كل بند P0 يستحق وثيقة خطة تنفيذ خاصة بـ schema migrations + UI mockups + test plan).
